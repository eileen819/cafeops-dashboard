# CafeOps Dashboard PRD

## 1. 프로젝트 개요

CafeOps Dashboard는 카페 운영자가 재고 상태, 입고 예정 정보, 발주 필요 품목, 판매 흐름을 한 화면에서 확인할 수 있도록 돕는 운영 대시보드입니다.

카페 매니저 경험에서 재고 확인, 입고 일정 확인, 발주 판단, 판매 흐름 파악이 반복적으로 발생한다는 점에 주목했습니다. 이 프로젝트는 운영자가 현재 매장 상황을 빠르게 파악하고, 다음 보충 주기까지 필요한 재고 수량을 기준으로 발주 여부를 판단할 수 있도록 돕는 것을 목표로 합니다.

---

## 2. 문제 정의

카페 운영에서는 품목 수가 많아질수록 현재 재고가 충분한지, 다음 입고 후 다음 보충 주기까지 버틸 수 있는지, 추가 발주가 필요한지 빠르게 판단하기 어렵습니다.

특히 단순히 마지막으로 확인한 재고 수량만 보면 실제 운영 상황을 반영하기 어렵습니다. 마지막 재고 확인일 이후 시간이 지나면 예상 사용량만큼 재고가 줄어들 수 있고, 그 사이 입고된 수량이 있다면 현재 재고 계산에 반영해야 합니다.

또한 발주한 물품이 다음날 입고되는 구조라면, 발주 수량은 단순히 다음 입고일까지 버티는 기준이 아니라, 다음 입고 후 다시 보충 가능한 시점까지의 예상 사용량을 기준으로 계산해야 합니다.

따라서 마지막 재고 확인일, 예상 하루 사용량, 마지막 확인 이후 입고 수량, 다음 입고 예정일, 다음 보충 가능일, 버퍼 재고를 함께 고려해 현재 예상 재고와 권장 발주 수량을 계산하는 구조가 필요합니다.

---

## 3. 목표

- 마지막 재고 확인일 이후의 예상 사용량을 반영해 현재 예상 재고를 계산합니다.
- 마지막 확인 이후 입고된 수량이 있다면 현재 예상 재고에 반영합니다.
- 다음 입고 예정일과 다음 보충 가능일을 기준으로 재고 판단 기간을 구분합니다.
- 다음 보충 주기까지 필요한 기준 재고를 계산합니다.
- 현재 예상 재고와 다음 보충 주기까지 필요한 기준 재고를 비교해 권장 발주 수량을 계산합니다.
- 재고 상태를 부족 / 정상 / 과잉으로 자동 분류합니다.
- 운영자가 상품을 등록하고 수정할 수 있게 합니다.
- 판매 흐름과 재고 부족 현황을 요약 카드와 차트로 확인할 수 있게 합니다.
- 주요 계산 로직은 UI와 분리해 테스트 가능한 함수로 작성합니다.

## 6. 핵심 사용자 흐름

1. 운영자는 대시보드에서 오늘의 판매 요약과 재고 부족 품목을 확인합니다.
2. 재고 관리 페이지에서 전체 품목을 검색, 필터, 정렬합니다.
3. 시스템은 마지막 재고 확인일과 예상 하루 사용량을 기준으로 현재 예상 재고를 계산합니다.
4. 마지막 확인 이후 입고된 수량이 있다면 현재 예상 재고에 반영합니다.
5. 시스템은 다음 보충 가능일까지 필요한 기준 재고를 계산합니다.
6. 시스템은 현재 예상 재고와 다음 보충 주기까지 필요한 기준 재고를 비교해 권장 발주 수량을 계산합니다.
7. 운영자는 권장 발주 수량이 있는 품목을 우선 확인합니다.
8. 필요한 경우 상품 정보를 등록하거나 수정합니다.

## 7. 핵심 계산 기준

### 7.1 현재 예상 재고

현재 예상 재고는 마지막으로 확인한 재고 수량에서 마지막 확인일 이후 예상 사용량을 차감하고, 그 사이 입고된 수량을 더해 계산합니다.

계산식:

`현재 예상 재고 = 마지막 확인 재고 + 마지막 확인 이후 입고 수량 - 예상 하루 사용량 * 마지막 확인일 이후 지난 일수`

음수가 되는 경우에는 `0`으로 처리합니다.

```ts
estimatedCurrentStock = Math.max(
  lastCheckedStock +
    receivedQuantitySinceLastChecked -
    expectedDailyUsage * daysSinceLastChecked,
  0,
);
```

---

### 7.2 다음 보충 주기까지 필요한 기준 재고

다음 보충 주기까지 필요한 기준 재고는 다음 보충 가능일까지 남은 일수와 예상 하루 사용량, 버퍼 재고를 기준으로 계산합니다.

여기서 다음 보충 가능일은 단순히 이번 입고일이 아니라, 이번 입고 후 다시 재고를 보충할 수 있는 시점을 의미합니다.

예를 들어 월요일에 발주하고 화요일에 입고되는 주기라면, 월요일에 발주할 수량은 화요일 입고 후 다음 주 화요일 입고 전까지 버틸 수 있는 수량을 기준으로 계산합니다.

계산식:

`다음 보충 주기까지 필요한 기준 재고 = 다음 보충 가능일까지 남은 일수 * 예상 하루 사용량 + 버퍼 재고`

```ts
requiredStockUntilNextCycleRestock =
  daysUntilNextCycleRestock * expectedDailyUsage + bufferStock;
```

---

### 7.3 권장 발주 수량

권장 발주 수량은 다음 보충 주기까지 필요한 기준 재고에서 현재 예상 재고를 뺀 값입니다.

계산식:

`권장 발주 수량 = 다음 보충 주기까지 필요한 기준 재고 - 현재 예상 재고`

음수가 되는 경우에는 `0`으로 처리합니다.

```ts
recommendedOrderQuantity = Math.max(
  requiredStockUntilNextCycleRestock - estimatedCurrentStock,
  0,
);
```

## 8. 계산 예시

| 항목                       | 값         |
| -------------------------- | ---------- |
| 마지막 확인 재고           | 30         |
| 마지막 재고 확인일         | 2026-04-25 |
| 오늘 날짜                  | 2026-04-28 |
| 예상 하루 사용량           | 5          |
| 마지막 확인 이후 입고 수량 | 0          |
| 다음 입고 예정일           | 2026-04-29 |
| 다음 보충 가능일           | 2026-05-06 |
| 버퍼 재고                  | 5          |

### 8.1 마지막 확인일 이후 지난 일수

`2026-04-28 - 2026-04-25 = 3일`

### 8.2 현재 예상 재고

`30 + 0 - 5 * 3 = 15`

현재 예상 재고는 `15`입니다.

### 8.3 다음 보충 가능일까지 남은 일수

`2026-05-06 - 2026-04-28 = 8일`

### 8.4 다음 보충 주기까지 필요한 기준 재고

`8 * 5 + 5 = 45`

다음 보충 주기까지 필요한 기준 재고는 `45`입니다.

### 8.5 권장 발주 수량

`45 - 15 = 30`

따라서 권장 발주 수량은 `30`입니다.

## 9. 재고 상태 기준

재고 상태는 현재 예상 재고, 다음 보충 주기까지 필요한 기준 재고, 최대 재고를 기준으로 분류합니다.

| 상태 | 기준                                                                                |
| ---- | ----------------------------------------------------------------------------------- |
| 부족 | 현재 예상 재고가 다음 보충 주기까지 필요한 기준 재고보다 작은 경우                  |
| 정상 | 현재 예상 재고가 다음 보충 주기까지 필요한 기준 재고 이상이고 최대 재고 이하인 경우 |
| 과잉 | 현재 예상 재고가 최대 재고를 초과하는 경우                                          |

### 판단 순서

1. 현재 예상 재고가 최대 재고를 초과하면 `과잉`으로 분류합니다.
2. 현재 예상 재고가 다음 보충 주기까지 필요한 기준 재고보다 작으면 `부족`으로 분류합니다.
3. 그 외에는 `정상`으로 분류합니다.

## 10. 성공 기준

- 사용자는 재고 부족 품목을 빠르게 파악할 수 있습니다.
- 현재 예상 재고는 마지막 재고 확인일, 예상 하루 사용량, 마지막 확인 이후 입고 수량을 기준으로 계산됩니다.
- 권장 발주 수량은 다음 보충 가능일까지 필요한 기준 재고와 현재 예상 재고를 비교해 계산됩니다.
- 재고 상태는 부족 / 정상 / 과잉으로 자동 분류됩니다.
- 상품 등록 / 수정 시 잘못된 입력은 폼 단계에서 차단됩니다.
- 재고 계산 로직은 UI와 분리되어 테스트로 검증됩니다.
- Mock API 기반으로 백엔드 없이 주요 화면 상태와 요청 / 응답 흐름을 확인할 수 있습니다.

## 11. 프로젝트 진행 기준

이 프로젝트는 화면을 먼저 만드는 방식이 아니라, 운영 기준을 기능 요구사항으로 바꾸는 방식으로 진행합니다.

카페 운영 흐름을 다음 단계로 나누어 구현합니다.

1. 재고 확인
2. 마지막 확인 이후 사용량 계산
3. 마지막 확인 이후 입고 수량 반영
4. 현재 예상 재고 계산
5. 다음 보충 가능일까지 필요한 기준 재고 계산
6. 권장 발주 수량 계산
7. 재고 상태를 화면에 표시
8. 계산 로직 테스트

이를 통해 단순 재고 테이블이 아니라, 운영자가 판단해야 하는 기준을 화면 상태와 계산 로직으로 구현하는 것을 목표로 합니다.
